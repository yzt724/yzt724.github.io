冈易云音乐

结合题目描述跟界面提示，这道题要求用户名jjkk114514，对应的激活码，也就是说，找到对应的流程是怎么运行的，使用用户名为jjkk114514顺序运行一遍，就可以得到激活码

猜测应该会有加密，找到了AES的相关内容

RijnDael_AES_LONG_455120是AES加密算法中的S盒，有三个函数引用了他，分别是

1.密钥扩展函数 (sub_4015C0)

作用：AES密钥扩展

  • 将原始16字节密钥扩展为轮密钥
  • 为每一轮AES加密生成所需的子密钥
  • 确保加密过程中每一轮都使用不同的密钥

2.行移位函数 (sub_401770)

作用：AES行移位操作

  • 对状态矩阵的每一行进行循环移位
  • 第一行：不移位
  • 第二行：左移1个字节
  • 第三行：左移2个字节
  • 第四行：左移3个字节
  • 增加加密的扩散性

3.列混合函数 (sub_401830)

作用：AES列混合操作

  • 对状态矩阵的每一列进行线性变换
  • 使用固定的多项式进行矩阵乘法
  • 进一步增强加密的扩散性和混淆性

RijnDael_AES_CHAR 和 RijnDael_AES_LONG 实际上指向同一个内存地址或包含相同的数据，它们指向同一个内存位置，是同一个AES S盒的不同类型表示

RijnDael_AES_LONG_inv_455020是 AES 加密算法中的逆 S 盒（Inverse S-Box），用于 AES 解密过程中的字节替换操作，它只在函数sub_401740中被调用

分析可知sub_426E80对用户名进行加密处理并验证激活码，其中处理步骤可分为

  1. 程序从配置文件读取用户名和激活码
  2. 对用户名进行MD5哈希
  3. 将用户名和MD5哈希组合
  4. 计算组合数据的SHA256哈希
  5. 分割SHA256哈希为左右各16字节
  6. 使用左16字节作为AES密钥，对用户输入的激活码进行加密
  7. 将加密结果与右16字节进行比较

解密需要使用SHA256后的右16字节为密文进行AES-128 ECB模式解密，得到激活码为c68b2adadacf92f286b73b8c6b6ca392

flower_dance

一个被修改过标志位的upx压缩壳，改回正确的标志位后再”upx -d”

十六进制 ASCII 码
55 50 58 30 UPX0
55 50 58 31 UPX1
55 50 58 32 UPX2
55 50 58 21 UPX!

发现两个同一类型的花指令（永真/永假跳转）

  1. xor ebx, ebx
  • 动作：将 ebx 寄存器与自己执行“异或”运算。
  • 结果：任何数与自己异或结果都是 0。所以这句执行完，ebx 的值必然是 0。
  • 副作用：它会修改 CPU 的标志寄存器（EFLAGS），因为结果是 0，所以 ZF（零标志位）会被设为 1。
  1. test ebx, ebx
  • 动作：对 ebx 执行“逻辑与”运算，但不保存结果，只影响标志位。
  • 结果：因为上一句刚把 ebx 变回 0，所以 0 & 0 的结果依然是 0。
  • 结论：这句其实是多余的（上一句已经改过标志位了），它在这里是为了再次确认 ZF = 1。
  1. jnz short near ptr byte_401256
  • 动作：jnz 的意思是 Jump if Not Zero（如果结果不为 0 则跳转，即当 ZF = 0 时跳转）。
  • 真相：由于已知 ZF 必然等于 1，所以这个跳转永远不会发生（这叫“永假跳转”）。
  • 陷阱：加壳者通常会在 00401256 这个地址放一个错误的字节（比如 0xE8 这种代表 Call 指令的开头）。反汇编器看到这句 jnz，会以为这里可能跳走，于是尝试去解析 401256 处的“假指令”，导致后面的代码显示全部乱掉。
  1. jz short loc_401257
  • 动作：jz 的意思是 Jump if Zero（如果结果为 0 则跳转，即当 ZF = 1 时跳转）。
  • 真相：因为 ZF 必然等于 1，所以这个跳转百分之百会发生（这叫“永真跳转”）。
  • 意义：它确保了程序能够“跳过”中间那个用来迷惑人的 401256 错误字节，直接跳到正确的代码起始点 00401257 继续运行。

ZeroTwo的权限

  1. 入口检查

程序首先检查输入的长度。如果不是 8 位，直接返回 0（失败）

  1. 陷阱排除 (Fake Flags)

程序会检查输入是否包含特定字符串（如 2025PCTF、PCTF 等，小彩蛋：YNPcz3与正确验证码前六位只有一个字母存在大小写的区别）

  • 如果命中这些字符串，validate 函数会返回特定的代码（如 36、37…）。

  • 在 UI 层的 switch 语句中，这些代码会引导程序调用 f() 函数，弹出一个假 Flag（例如 PCTF{this_is_fake_flag_1}）

  1. 核心生成：E(input) 变换

如果输入是普通的 8 位字符串，程序会调用复杂的 E(input) 函数。这个函数对输入进行了多步数学和位运算变换：

  1. XOR 混淆：输入与一组动态生成的密钥进行异或
  2. 循环左移：字节级别的数据循环移动
  3. 代数运算：进行非线性的加减乘除混淆
  4. S 盒置换：利用硬编码的 NP 数组，将字节的高低 4 位进行映射替换（这是分组加密的典型特征）
  5. 字节翻转：交换高低 4 位
  • 结果：生成一个 8 字节的加密数组 E

  1. 目标生成：R() 函数

程序同时调用 R() 函数。这个函数利用一组硬编码的种子数据（rcSeed），通过简单的异或运算，生成一个固定的 8 字节目标数组

  1. 最终比对

调用 eq(E, R) 函数，比对 E(input) 的结果是否与固定的 R() 完全一致。这是一个经典的恒定时间比较（Constant-Time Comparison）函数，功能上它和 Arrays.equals(a, b) 是一样的，但它可以防范计时攻击

普通比较的函数：如果密码的第一位就错了，程序返回极快；如果密码前 n-1 位都对，只有第 n 位错了，程序返回会稍微慢一点点。可以通过精密测量程序返回的时间，逐位爆破出密码

eq函数：不论密码是对是错，也不论是哪一位错，它都必须完整地跑完整个循环，它没有 break 或 return false 在循环中间，它利用 r |= ... 把所有的比较结果都累计起来，最后才判断

结果：对于相同长度的输入，函数的执行时间是恒定的，无法通过“计时”来推测密码

真正的 Flag 并不是预先存储在 App 里的字符串，而是通过算法动态解密出来的。只有当上述验证通过（eq(E, R) 返回 true）时，解密流程才会被正确触发

  1. 密钥准备：lastEnc

当验证通过时，程序会将 lastEnc（一个私有的静态字节数组）赋值为成功的 E(input) 结果（也就是 R() 的结果）

  • 关键点：R() 的结果就是解密真正 Flag 的密钥
  1. 校验准备：gate

程序会根据 E（即 R）的内容，手动计算一个校验位 gate：

并将这个值存入私有静态变量 gate 中

  1. 执行解密函数 F()

当 UI 层收到 validate 返回的 1 时，会调用 F() 函数。F() 的执行逻辑如下：

  1. 依赖检查：检查 lastEnc 是否为空，且手动重新计算的 g2 是否等于之前存入的 gate。如果检查失败，解密终止，返回“验证失败”。
  2. 密文获取：获取硬编码的十六进制字符串 FC_HEX 作为密文。
  3. 密钥流生成：利用 lastEnc（密钥）和自定义的 keystream 算法，生成一组伪随机的密钥流（ks）。
  4. 算法解密：
    • 利用 NPinv（NP 数组的逆向映射）对密文进行逆置换。
    • 与密钥流 ks 进行异或。
    • 进行 swapN（高低 4 位交换）。
    • 最后与一个基于循环索引的动态值 (i * 31 + 93) 进行异或。
  • 结果：解密出的字节数组被转换为 UTF-8 字符串，这就是最终的真正的 Flag。
