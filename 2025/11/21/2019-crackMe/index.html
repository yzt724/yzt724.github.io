main函数

经过分析可知，真正的加密过程是在main函数发生异常后，系统执行的，而不是以call直接调用，所以只看汇编是找不到真正执行的函数

main函数最后call sub_411136是在比较两个字符串，也就是说执行完一系列加密后，最终回到main函数进行比较，返回right或者wrong

接着分析，当main发生异常后，系统会依次执行哪些函数，这些函数各有什么作用，首先分析main函数中的

sub_41100F函数

查看交叉引用看是哪里生成了轮密钥数组，交叉引用到

Handler_0函数

发现了该函数同时存在

  1. if ( *a1 == -1073741819 )，这个判断条件之前就出现过
  2. 一直分析sub_411172函数，发现最后也调用了sub_4119A0

所以这里的加密逻辑肯定也是程序会真正执行到的，分析Handler_0函数，注意粉色那行，把程序的 UEH 设置为了TopLevelExceptionFilter函数

1.首先把where_are_u_now?字符串赋值给v2

2.sub_411172及该函数接下来调用的函数主要对”where_are_u_now?”进行处理，处理完得到轮密钥数组dword_41A218，该数组在之前sm4加密函数里也引用到了

3.TopLevelExceptionFilter函数及该函数接下来调用的函数对Str2（即main最后比较时的密文）进行了换位操作，对sm4得到的密文进行魔改base64加密

目前已知流程

1.使用轮密钥dword_41A218对明文进行sm4加密

2.比较

3.对密钥where_are_u_now?处理得到轮密钥dword_41A218

4.密文轮换

5.base64魔改加密

这些流程只存在三个地方，main函数，handler_0函数中的sub_411172函数，handler_0函数中的TopLevelExceptionFilter函数

要搞清楚它们执行时的正确顺序，就需要知道程序在发生异常时是怎么跳转执行的，接着往上分析交叉引用

其中 AddVectoredExceptionHandler(0, (PVECTORED_EXCEPTION_HANDLER)Handler);的含义是：程序发生异常的时候，首先调用Handler函数

第1个参数如果是 0 → 这个 Handler 以最高优先级插入到链表最前面

第2个参数 → 用户自己写的异常处理函数地址

这就是VEH

VEH SEH UEH简单介绍

类型 名称 注册函数 特性
VEH Vectored Exception Handler AddVectoredExceptionHandler 优先级最高，最先执行
SEH Structured Exception Handler FS:[0] 链、try/except 每次异常都捕获、按链表级别执行
UEH Top Level Unhandled Exception Filter SetUnhandledExceptionFilter 只处理未捕获异常，最后一道屏障

UEH 的返回值含义（MSDN）：

返回值 意义
EXCEPTION_EXECUTE_HANDLER (1) 进入 __except 处理逻辑
EXCEPTION_CONTINUE_SEARCH (0) 继续按 SEH 链查找处理函数（最终可能退出程序）
EXCEPTION_CONTINUE_EXECUTION (-1) 继续执行异常指令

在这道题中

作用

UEH 密文str2轮换，base64索引表凯撒加密，base64魔改加密，返回值-1，回到main函数继续执行
